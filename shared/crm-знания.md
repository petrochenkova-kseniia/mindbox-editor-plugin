# CRM-знания для работы с кейсами Mindbox

Дистилляция из 48 уроков школы CRM-маркетологов Mindbox.
Используй при анализе фактуры кейса, формировании структуры, уточняющих вопросах.

---

## 1. Данные о клиентах

**Типы данных:**
- **Идентификационные** — email, телефон, имя, ИД в приложении, ИД в офлайн-кассе
- **Транзакционные / поведенческие** — история покупок, просмотры сайта, активность в каналах (открытия, клики)
- **Дополнительные** — психографика, интересы, анкетные данные, данные из кабинета лояльности

**CRM vs CDP:**
- CRM — управляет прямыми взаимодействиями и продажами; исторически = база контактов
- CDP (Customer Data Platform) — собирает данные из всех источников (веб, приложение, CRM, офлайн-кассы) в единый профиль клиента
- Единый профиль клиента (ЕПК) = демография + контакты + история покупок + поведение на сайте/в приложении + история коммуникаций + история начислений/списаний бонусов

**Реальное время vs батч:**
- Real-time нужен для триггеров (брошенная корзина, price drop, поведенческие события)
- Батч (ночные задачи) достаточен для аналитики и плановых промо-кампаний

**Качество данных:**
- Дубликаты в базе → ЕПК не формируется → клиент получает одно и то же письмо дважды
- Неидентифицированные клиенты (офлайн без email) → не получают CRM-коммуникации, но их транзакции должны учитываться в аналитике
- Мастер-данные (золотая запись клиента) — единственная каноническая версия профиля, из которой все системы берут данные

## 2. Каналы

### Классификация

| Бесплатные (стоимость ≈0) | Платные |
|---|---|
| Email | SMS (~6 ₽/сообщение) |
| Mobile push (нужно приложение) | WhatsApp (~2–5 ₽/сообщение) |
| Web push | |
| Telegram-бот / VK-бот | |
| On-site (поп-апы, баннеры) | |
| In-app | |

### Порядок запуска каналов
1. **Email + mobile push** — базовый старт; бесплатно, максимальный охват
2. **SMS в каскаде + on-site поп-апы** — через месяц-два после стабилизации email
3. **Telegram-бот / VK-бот** — только если есть активное сообщество (50+ новых подписчиков/мес, 2–3 поста/нед в группе)
4. **Web push** — последний приоритет: низкий % согласий, много блокировщиков, высокий unsubscribe

### Минимальные условия для запуска

| Канал | Условие |
|---|---|
| Email | База от ~1000 адресов; домен прогрет |
| Mobile push | Есть приложение с активными установками |
| SMS | Средний чек > ~300–500 ₽ (иначе стоимость 6 ₽/SMS не окупается) |
| WhatsApp | API-интеграция через официального партнёра; только при высоком LTV |
| Омниканальность | 2+ каналов, база 10k+, выделенный CRM-маркетолог, CDP |

### Особенности WhatsApp
- Требует официального Business API (через партнёров — Infobip, Waba360 и др.)
- Нельзя рассылать рекламные сообщения без предварительного opt-in клиента
- Стоимость выше email/push → использовать в конце каскада или для VIP-сегментов
- Высокий open rate (~70–80%) — но это не бизнес-метрика без конверсии

## 3. Каскад и омниканальность

### Мультиканальность vs Омниканальность

| | Мультиканальность | Омниканальность |
|---|---|---|
| Базы данных | Разрозненные (email-база отдельно, SMS-база отдельно) | Единый профиль клиента (CDP) |
| Коммуникации | Независимые по каналам, клиент получает дубли | Каскадные цепочки, исключение отреагировавших |
| Результат | Клиент получает 3 одинаковых сообщения | Клиент получает одно сообщение в нужном канале |
| Аналитика | Метрики по каналам разрознены | Единая аналитика по клиенту |

### Компоненты единого профиля клиента (ЕПК)
- Контактные данные (email, телефон, ID приложения)
- Демография (имя, дата рождения, пол, город)
- История транзакций (онлайн + офлайн)
- Поведение на сайте и в приложении
- История коммуникаций (что получал, что открывал, на что кликал)
- Данные программы лояльности (бонусы, статус)

### Логика каскада для брошенной корзины

| Шаг | Канал | Условие | Тайминг |
|---|---|---|---|
| 1 | Mobile push | Есть приложение | Через 1 час |
| 2 | Email | Не открыл пуш | Через 3 часа от события |
| 3 | SMS | Не открыл email | Через 12 часов от события |
| 4 | WhatsApp | Есть номер, не среагировал | Через 24 часа от события |

> Правило: следующее сообщение — только если предыдущее не сработало.

### Экономика каскада (пример, 10 000 клиентов)
- Без каскада: 60 000 ₽ (SMS всем)
- С каскадом: 12 000 ₽ (SMS только тем, кто не открыл email — ~20% от базы)
- Экономия: 48 000 ₽ на одной рассылке

### Когда омниканальность НЕ нужна
- Только один канал коммуникации
- База < 10 000 контактов
- Нет выделенного CRM-маркетолога
- Нет аналитики и инфраструктуры для сравнения каналов

### Ключевая метрика омниканальности
**RPE (Revenue per Email)** — растёт при правильной синхронизации каналов. Если каскады настроены правильно, RPE растёт без роста затрат.

## 4. Качество базы

### Нормы метрик

| Метрика | Норма | Тревожный сигнал |
|---|---|---|
| Delivery Rate | >99% | <98% |
| Bounce Rate (hard+soft) | 1–2% | >3% |
| Spam complaints | <0,1% (Gmail), <1% (Mail.ru) | >0,3% Gmail — критично |
| Unsubscribe Rate | <0,5% | >1% |
| CTOR (Click-to-Open Rate) | 10–20% | <5% |

### Что ухудшает репутацию домена
- Высокий bounce rate (письма на несуществующие адреса)
- Жалобы на спам
- Отправка по неактивным адресам (90+ дней без открытия)
- Резкий рост объёма отправок без прогрева
- Использование одного IP-адреса для больших объёмов без прогрева

### Работа с доставляемостью
- Мониторинг: **Google Postmaster Tools** (репутация домена у Gmail), **Mail.ru Postmaster** (репутация у Mail.ru)
- Техническая настройка: SPF, DKIM, DMARC-записи → без них письма уходят в спам
- Прогрев домена: начинать с малых объёмов (1000/день), постепенно увеличивать за 4–6 недель
- Dedicated IP vs shared IP: при объёмах 100k+/мес — dedicated IP, иначе shared достаточен

### Работа с неактивными контактами

| Сегмент | Определение | Что делать |
|---|---|---|
| Активные | Открывал письмо в последние 90 дней | Отправлять всё |
| Засыпающие | 90–180 дней без активности | Снизить частоту, попробовать реактивацию |
| Спящие | 180+ дней без активности | Реактивационная кампания → при отсутствии реакции → удаление из активной базы |

- Спящие деградируют доставляемость всей базы
- Удаление из рассылок ≠ удаление из базы данных — оставлять в CRM, только исключать из активных рассылок

## 5. Сегментация

### По стадии жизненного цикла (воронка)
- Лид (есть контакт, нет покупки)
- Новый (1 покупка, < 30 дней)
- Развивающийся (2–3 покупки)
- Лояльный (4+ покупок, стабильная частота)
- Под угрозой оттока (давно не покупал, но были регулярные покупки)
- Спящий (нет покупки 90–180+ дней)

### RFM-сегментация

| Параметр | Что значит | Как применять |
|---|---|---|
| R (Recency) | Когда последний раз купил | Недавние → активная коммуникация; давние → реактивация |
| F (Frequency) | Как часто покупает | Высокая → приоритетный сегмент; низкая → стимул повторной покупки |
| M (Monetary) | На какую сумму купил | Высокий → VIP-обслуживание; низкий → апсейл |

> RFM применяют для сегментации, приоритизации коммуникаций и выявления VIP-сегмента.

### По поведению
- Интересы по категориям (на основе просмотров и покупок)
- Активность в каналах: открывает email / не открывает / кликает / не кликает
- Поведение на сайте: был в разделе, добавил в корзину, просматривал конкретный товар

### По демографии
- Пол, возраст, наличие детей, семейное положение
- Применять с осторожностью: пол часто не указан явно → определяется по поведению или не используется

### По географии
- Город / регион
- Зона доставки (актуально для доставки еды, офлайн-ритейла с разными зонами)

### По каналу коммуникации
- Есть email / нет email
- Есть mobile push / нет (нет приложения или нет разрешения)
- Есть phone / нет phone
- Предпочтительный канал (определяется по активности)

## 6. Политика коммуникаций

### Приоритеты (от высшего к низшему)
1. **Сервисные / транзакционные** — подтверждение заказа, статус доставки, смена пароля, чек → отправлять всегда, не учитываются в лимитах частоты
2. **Триггерные по поведению** — брошенная корзина, реактивация, price drop → если нет конфликта с другими
3. **Массовые промо** — акции, дайджесты, контент-рассылки → последний приоритет, учитываются в дневном лимите

### Управление частотой
- Лимит: не более 1–2 промо-писем в неделю (зависит от отрасли и базы)
- Global suppression: если клиент получил письмо A, не отправлять письмо B в течение X дней
- Транзакционные не учитываются в лимите частоты
- Сегментация по усталости: если клиент не открывает 5+ последних писем → снижать частоту или исключить из промо

### Pressure management (управление давлением)
- Цель: максимизировать relevance, минимизировать irritation
- Инструмент: на уровне CDP — настройка глобальных ограничений по частоте
- Метрика здоровья: Unsubscribe Rate < 0,5%; если растёт → давление слишком высокое

## 8. Базовые сценарии (механики)

### Welcome-цепочка
- 3–5 писем в течение 1–2 недель
- Задачи: онбординг продукта/сервиса, знакомство с программой лояльности, стимул первого заказа
- Персонализация: не отправлять то, что клиент уже сделал (подписан в соцсетях → не зовём; уже купил → переходит в следующий сегмент)
- Пример результата: +35,5% конверсии в покупку vs без цепочки (Jivo)
- Ошибка: включать всех новых клиентов без исключений — нужно исключить тех, кто уже купил или уже знаком с сервисом

### DOI (Double Opt-In)
- Задача: подтверждение email для очистки базы
- Нужен компании, а не клиенту → мотивация: давать бонус/скидку за подтверждение
- Click rate DOI: 15–30% от отправленных
- Без DOI: в базу попадают несуществующие адреса → высокий bounce → репутация домена падает

### Брошенный просмотр
- Стандарт для e-com; внедряется после welcome и DOI
- Показывает просмотренные товары + напоминание
- Тайминг: 2–4 часа после сессии просмотра
- Требует трекинга поведения на сайте (события в CDP)

### Брошенная корзина
- В напоминании: список товаров, напоминание о дефицитности («осталось 2 штуки»)
- Уведомление о снижении цены товаров в корзине — отдельная механика (price drop alert)
- Конверсия из письма: 2–5% (среднее; высокое значение — специфика конкретного сегмента)

### Брошенная оплата
- Самый горячий триггер: клиент уже выбрал товар и начал оплату
- Ссылка прямо на страницу оплаты (deeplink)
- Можно добавить персональную скидку для мотивации завершить покупку

### Реактивация
- Win-back оффер: скидка, бонусы, специальное предложение
- Тайминг: 90–180 дней без покупки (зависит от частоты покупок в категории)
- Сценарий: 2–3 письма с нарастающим оффером → если нет реакции → вывод из активной базы
- Результат: 5–15% реактивированных

### Напоминание о повторной покупке (replenishment)
- Строится на частоте покупок в категории (например, косметика → раз в 2–3 месяца)
- Событие: прошло X дней с последней покупки из категории
- Точность персонализации зависит от качества данных о частоте потребления

### Коммуникации программы лояльности
- Уведомление о начислении баллов → транзакционное, отправлять всегда
- Уведомление о сгорании баллов → триггерное, за 7–14 дней до сгорания
- Апгрейд статуса → поздравление с новым статусом + объяснение привилегий
- NPS-опрос после покупки → через 1–3 дня после доставки

## 9. Продвинутые коммуникации

### Уровни персонализации

| Уровень | Что персонализируется | Пример |
|---|---|---|
| Базовый | Имя, город | «Привет, Иван!» |
| Средний | Канал, тайминг | SMS только тем, кто не открывает email |
| Продвинутый | Контент | Показывать категории, которые клиент смотрел |
| Экспертный | Оффер | Индивидуальная скидка на основе LTV и истории покупок |
| ML-уровень | Всё | Предсказание лучшего времени, канала, контента, оффера |

### Персонализация по личному циклу потребления
- Вместо общего тайминга (90 дней) — индивидуальный на основе истории покупок клиента
- Пример: BetBoom — реактивация по персональному циклу потребления вместо фиксированного тайминга → Retention ×2,4
- Требование: данные о транзакциях + модель предсказания следующей покупки

### ML-коммуникации: предпосылки для внедрения

Все четыре условия должны выполняться одновременно:
1. **CDP с единым профилем клиента** — данные собраны, дедуплицированы, обогащены
2. **Качество данных** — достаточная история транзакций для обучения модели (12+ месяцев, тысячи клиентов)
3. **IT-интеграции** — CDP может отправлять события в систему рассылок в real-time
4. **Data science ресурс** — команда или партнёр, способный обучать и поддерживать модели

### Примеры ML-результатов
- М.Видео: ML-персонализация коммуникаций → +1,5 млрд ₽/мес инкрементальной выручки
- Требует контрольных групп для доказательства инкрементальности

### Что НЕ персонализировать без данных
- Не использовать пол, если он не подтверждён
- Не делать персональные скидки без понимания price sensitivity клиента
- Не запускать ML без достаточной истории данных → модель даёт нерелевантные рекомендации

## 10. Аналитика

### Воронка коммуникации (8 этапов)

1. Целевая аудитория (кому потенциально отправлять)
2. Доступная аудитория (у кого есть контакт в нужном канале)
3. Подписанные (дали согласие на коммуникации)
4. Отправлено
5. Доставлено
6. Открыто
7. Кликнуто
8. Конверсия (заказ, покупка, целевое действие)

**Правило анализа:** разбирать снизу вверх — этапы ближе к конверсии дают максимальный эффект при улучшении.

**Где ищем узкое место:**
- Низкая доставляемость → проблема с базой или репутацией домена
- Низкий OR → проблема с темой или сегментом
- Низкий CTR → проблема с контентом или оффером
- Низкая конверсия из клика → проблема с лендингом или оффером

**Сегментация воронки по реакции:**
- Не открыли → проблема с темой или доставляемостью
- Открыли, не кликнули → проблема с контентом или оффером
- Кликнули, не купили → проблема с лендингом или процессом оформления

### Когортный анализ

**Когорта** = группа клиентов с одинаковым признаком (обычно — дата первой покупки).

**Таблица удержания (Retention Rate):**
- Строки = когорты по месяцам
- Столбцы = периоды после первой покупки (месяц 1, 2, 3...)
- Значение в ячейке = % клиентов когорты, купивших в этом периоде
- Нормальное поведение: удержание снижается со временем, но стабилизируется

**Churn Rate (отток):**

`Churn = (Активные прошлого периода + Новые прошлого периода) − Активные текущего периода`

- Активный клиент = совершивший покупку в периоде
- Рост churn → проблема с удержанием или сезонность

**Рост покупательской базы:**

`Прирост = Новые клиенты − Отток`

- Цель: прирост близко к 0 или положительный (база не «вытекает»)
- Если отток > новых → база сжимается

**Когортный анализ выручки:**
- Сравниваем LTV когорт: становятся ли новые когорты ценнее или дешевле старых
- Рост LTV новых когорт = маркетинг привлекает более качественных клиентов

### Инкрементальный эффект (контрольные группы)

**Ключевой принцип:** метрики рассылок (OR, CTR, CR) показывают качество коммуникации, но НЕ доказывают бизнес-эффект. Нужна контрольная группа.

**Контрольная группа:**
- 5–20% аудитории, которой НЕ отправляют коммуникацию
- Сравниваем выручку / конверсию / LTV группы с коммуникацией vs контрольной
- Разница = инкрементальный эффект

**Ошибка без контрольной группы:**
- «Наши письма принесли 30% выручки» — но эти клиенты могли купить сами, без письма
- Реальный инкрементальный эффект может быть 5–15% от заявленного

### UTM-разметка

| Параметр | Обязателен | Значение |
|---|---|---|
| `utm_source` | Да | Источник: `email`, `sms`, `push`, `whatsapp` |
| `utm_medium` | Да | Тип: `trigger`, `promo`, `transactional`, `digest` |
| `utm_campaign` | Да | Название кампании или автосценария |
| `utm_content` | Нет | Версия письма (для AB-теста), кнопка |
| `utm_term` | Нет | Сегмент или другой доп. признак |

**Правила UTM:**
- Без UTM — трафик из email попадёт в «прямой» или «реферальный» → атрибуция ломается
- Единый стандарт UTM для всей компании — иначе сравнение каналов невозможно
- Автоматизировать генерацию UTM в CDP/ESP — не ставить вручную

## 11. Метрики каналов (вовлечённость)

### Email

| Метрика | Формула | Норма | Что показывает |
|---|---|---|---|
| Delivery Rate | Доставлено / Отправлено × 100% | >99% | Качество базы и репутации домена |
| Open Rate (OR) | Открыто / Доставлено × 100% | 15–25% | Релевантность темы и сегмента |
| CTOR | Кликнуто / Открыто × 100% | 10–20% | Качество контента и CTA |
| CTR | Кликнуто / Доставлено × 100% | 2–5% | Общая эффективность рассылки |
| Unsubscribe Rate | Отписки / Доставлено × 100% | <0,5% | Усталость от рассылок |
| Complaint Rate | Жалобы / Доставлено × 100% | <0,1% (Gmail) | Нарушение ожиданий клиента |

### SMS

| Метрика | Норма | Комментарий |
|---|---|---|
| Delivery Rate | >95% | Зависит от оператора; ниже у коротких номеров |
| CTR (из доставленных) | 2–5% | Без ссылки измеряется по промокоду |

### Push (mobile и web)

| Метрика | Норма | Комментарий |
|---|---|---|
| Delivery Rate | 60–80% (mobile push) | Часть пользователей удалила приложение или отозвала разрешение |
| CTR (mobile push) | 2–8% | Зависит от сегмента и персонализации |
| CTR (web push) | 1–3% | Ниже из-за широкого охвата |

**Важно:** метрики вовлечённости (OR, CTR) показывают качество коммуникации, но НЕ являются бизнес-метриками. Для оценки влияния на бизнес нужны конверсия и инкрементальная выручка с контрольными группами.

## 12. Денежные метрики

### Conversion Rate (CR)

`CR = Количество заказов / Количество доставленных сообщений × 100%`

- Для длинного цикла сделки (B2B, мебель, недвижимость) — CR не подходит как метрика рассылок: между письмом и покупкой — месяцы
- Для e-com — считать в окне 3–7 дней после отправки
- Контрольная группа обязательна для доказательства инкрементальности CR

### Average Order Value (AOV)

`AOV = Выручка / Количество заказов`

- Растёт при правильном апсейле и кросс-сейле в коммуникациях
- Падение AOV при росте транзакций — возможно, скидочная стратегия «обесценивает» товар

### ROMI (Return on Marketing Investment)

`ROMI = (Инкрементальная выручка − Расходы) / Расходы × 100%`

**Расходы включают:**
- Стоимость ESP (платформы рассылок, тариф Mindbox)
- Стоимость контента и дизайна
- Зарплата CRM-команды (пропорционально)
- Стоимость скидок и бонусов (промокоды — прямые расходы)

**Главная ошибка в расчёте ROMI:** считать всю выручку от тех, кто получил письмо, а не инкрементальную. Пример: 100 человек купили после письма на 1 000 000 ₽, но 80 из них купили бы без письма → инкрементальная выручка 200 000 ₽, а не 1 000 000 ₽.

### RPE (Revenue per Email)

`RPE = Инкрементальная выручка / Количество отправленных писем`

- Ключевая метрика эффективности email-канала
- Растёт при улучшении сегментации и релевантности
- Триггеры обычно дают RPE в 3–10× выше промо

### Контентные письма и их метрики
- Bosco: 43% всей email-выручки приходит от контентных рассылок (не промо)
- Метрика для контента: не открытия, а переходы на сайт и покупки после визита
- Нельзя сравнивать промо и контент по OR: у контента OR ниже, но LTV клиентов выше

## 13. AB-тесты

### Что тестировать по каналам

**Email:**
- Тема письма (основной элемент влияния на OR)
- Прехедер (текст после темы)
- Время отправки
- Контент (макет, структура, длина)
- CTA (текст кнопки, цвет, позиция)
- Персонализация (имя, товарные рекомендации)
- Сегмент отправки

**SMS:**
- Первая фраза (всё, что видит клиент до «Читать дальше»)
- CTA (ссылка vs промокод)
- Тайминг

**Push (mobile/web):**
- Заголовок
- Текст уведомления
- Изображение
- Тайминг
- Сегмент

**Автосценарии (триггеры):**
- Тайминг запуска (1 час vs 2 часа для брошенной корзины)
- Количество шагов в цепочке
- Контент каждого шага
- Наличие скидки в последнем письме

### Когда НЕ тестировать
- Аудитория меньше минимального размера
- Есть очевидная ошибка в текущей версии — просто исправить
- Горящий период (Чёрная пятница, крупная акция) — риск потерять выручку на тестовой группе
- Нет аналитики и инфраструктуры для замера
- Нет ресурсов в команде для анализа результатов

### Формула гипотезы

`«Если [конкретное изменение], то [метрика вырастет с X до Y], потому что [обоснование]»`

Пример хорошей гипотезы:
> «Если добавить в тему письма имя получателя, то OR вырастет с 18% до 22%, потому что персонализация повышает релевантность»

Пример плохой гипотезы:
> «Если изменить цвет кнопки, может быть, станет лучше»

### Приоритизация гипотез

**Метод ICE:**

| Критерий | Шкала (1–10) | Что оценивать |
|---|---|---|
| Impact (влияние) | 1–10 | Насколько сильно улучшит ключевую метрику |
| Confidence (уверенность) | 1–10 | Насколько уверены в гипотезе (есть ли данные) |
| Ease (простота) | 1–10 | Насколько легко реализовать |

`ICE Score = Impact × Confidence × Ease`

**Метод RICE:**

| Критерий | Единица |
|---|---|
| Reach (охват) | Кол-во клиентов, которых затронет |
| Impact (влияние) | Оценка 1–10 |
| Confidence (уверенность) | % уверенности (0–100%) |
| Effort (усилия) | Человеко-месяцы |

`RICE Score = (Reach × Impact × Confidence%) / Effort`

**MoSCoW** — для приоритизации задач разработки (не гипотез):
- Must Have / Should Have / Could Have / Won't Have

### Правила выбора метрик для теста

1. **Первичная метрика** — одна, главная, напрямую связана с гипотезой (тестируем тему → первичная метрика OR)
2. **Вторичные метрики** — 2–3 дополнительных, для контроля побочных эффектов
3. **Guardrail-метрики** — метрики, которые нельзя ухудшать (Unsubscribe Rate, жалобы на спам)
4. **Правило одного изменения:** менять только один элемент за тест → иначе неясно, что повлияло

### Объём выборки

| Метрика | Мин. аудитория на вариант |
|---|---|
| Open Rate (тема письма) | 1000–2000 |
| CTOR (контент) | 1500–3000 |
| Conversion Rate (конверсия) | 5000–10 000 |

- Уровень значимости: p-value < 0,05 (95% confidence)
- Меньше выборки → выше риск случайного результата → нельзя делать выводы

### Типы тестов

| Тип | Описание | Когда применять |
|---|---|---|
| AB-тест | Два варианта (A и B) | Стандартный тест одного изменения |
| ABC-тест | Три варианта (A, B, C) | Когда несколько равнозначных гипотез |
| AA-тест | Два одинаковых варианта | Проверка корректности системы тестирования (должны показать 0% разницы) |
| AAB-тест | AA-проверка + тест одновременно | Валидация системы и тест одновременно |

## 14. CRM-стратегия

### Три уровня стратегии

```
Бизнес-стратегия (куда растём, какая выручка)
    ↓
Маркетинговая стратегия (как привлекаем и удерживаем клиентов)
    ↓
CRM-стратегия (как коммуникации влияют на LTV и удержание)
```

CRM-стратегия не существует сама по себе — она вытекает из бизнес-целей.

### 4 элемента CRM-стратегии

1. **Аналитика клиентской базы** — RFM-анализ, когорты, качество данных, LTV по сегментам
2. **Цели и KPI** — связаны с P&L, конкретные метрики (инкрементальная выручка, Retention Rate, LTV)
3. **Гипотезы и роадмап** — приоритизированный список того, что тестируем и в каком порядке
4. **Измерение ROI** — инкрементальный подход, контрольные группы, регулярная отчётность

### 5 типичных ошибок в CRM-стратегии

1. **CRM = рассылки** — нет стратегии, только тактика. Решение: начать с анализа базы и целей
2. **Нет связи с P&L** — непонятно, как CRM влияет на прибыль. Решение: связать KPI с выручкой и маржой
3. **Нет гипотез и тестов** — работают по шаблону, не оптимизируют. Решение: роадмап из гипотез
4. **Нет пересмотра стратегии** — написали раз, не обновляют. Решение: ревью каждый квартал
5. **Массовые скидки без анализа** — скидочная стратегия приучает к промо; при отмене акций: +25% к оттоку

### Правильный расчёт ROI: чеклист

- [ ] Использовать инкрементальную выручку, а не всю выручку от получателей
- [ ] Сформировать контрольную группу (5–20% от аудитории)
- [ ] Включить в расходы: ESP + контент/дизайн + команда + скидки/промокоды
- [ ] Считать за период, а не за одну рассылку
- [ ] Учесть сезонность при сравнении периодов
- [ ] Не сравнивать активных покупателей с пассивными (selection bias)
- [ ] Проверить, что контрольная группа репрезентативна (такой же состав, как тестовая)
- Формула: `(Выручка с коммуникацией − Выручка контрольной группы) − Расходы`

### 5 признаков зрелой CRM-стратегии

1. Есть связь между CRM KPI и P&L компании
2. Все решения сопровождаются гипотезой с ожидаемым эффектом
3. Регулярно проводятся AB-тесты с контрольными группами
4. Стратегия пересматривается раз в квартал на основе данных
5. CRM-команда участвует в продуктовых обсуждениях

## 15. Красные флаги в фактуре кейса

| Ситуация | Вопрос |
|---|---|
| OR <15% | Проблема с доставляемостью или темой? Как считают? |
| CTR <2% | Контент не попадает в сегмент или оффер слаб? |
| Unsubscribe >1% | Что изменили в рассылках? Резкое учащение? |
| CR <0,5% из клика | Проблема с лендингом или оффером? |
| Нет контрольной группы | Как атрибутировали результат? Инкрементальность доказана? |
| ROI без учёта скидок | Скидки считали как расход? |
| Сравнение «было/стало» без периода | За какой период? Сезонность учитывали? |
| Метрика «выручка от CRM» | Как считали атрибуцию — last click, first click, инкремент? |
| Очень высокий конверт | Нетипично высокое значение — как считали, есть ли контрольная группа? |
| Высокий OR без конверсии | OR ≠ выручка; CTOR и CR смотрели? |
| Нет периода измерения | Ни одна метрика не имеет смысла без временного окна |
| Выборка < 1000 для AB-теста | Статистически незначимо; нельзя делать выводы |

## 16. Типичные результаты по механикам (ориентиры)

Используй для проверки метрик в фактуре. Если результат сильно отличается — уточни методологию.

| Механика | Типичные результаты |
|---|---|
| Welcome-цепочка → конверсия в первую покупку | +20–40% vs без цепочки |
| Брошенная корзина → конверсия из письма | 2–5% |
| Реактивация → возврат | 5–15% реактивированных |
| DOI → click rate | 15–30% |
| Персонализация по личному циклу → удержание | ×2,4 (BetBoom) |
| ML-персонализация → инкрементальная выручка | +1,5 млрд ₽/мес (М.Видео) |
| Омниканальность → выручка от CRM | +35% (Снежная Королева) |
| Омниканальность → расходы на CRM | −1,5× после оптимизации каскадов |
| Контентные письма → доля в email-выручке | 43% (Bosco) |
| Брошенная корзина → конверсия (ресейл) | 29,49% (Second Friend Store — высокий, специфика ниши) |

## 17. CJM и базовые сценарии

**CJM (Customer Journey Map) не строит CRM-маркетолог в одиночку:**
- Данные рассылок + CRM → даёт маркетолог
- Аналитика сайта → продуктовые аналитики
- Барьеры и боли → исследователи + поддержка
- Полная CJM = командная работа

**Этапы CJM:**
Интерес → Выбор → Первое действие (регистрация/установка) → Конверсия → Повторная конверсия

**Важно:** барьеры на этапах — не всегда проблема в коммуникациях.
Пример: клиенты не переходили к повторной покупке не из-за плохих писем, а из-за скрытой кнопки «Мои заказы» в личном кабинете. CRM мог слать письма вечно — проблема была в UX.

**Какие механики закрывают какие этапы:**

| Этап CJM | Механика |
|---|---|
| Лид → регистрация | Welcome + DOI |
| Регистрация → первая покупка | Welcome-цепочка с онбордингом |
| Просмотр → покупка | Брошенный просмотр, брошенная корзина |
| Разовый → повторный | Напоминание о повторной покупке, NPS + возврат |
| Активный → под угрозой | Реактивационная цепочка |
| Под угрозой → спящий | Win-back оффер, последняя попытка реактивации |
