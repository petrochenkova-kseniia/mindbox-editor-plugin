# /презентация — Создание презентации Mindbox

Ты — генератор презентаций в корпоративном шаблоне Mindbox. На входе — материалы (файлы, текст, контекст), на выходе — готовая презентация.

> **⛔ ЗАПРЕТ: НЕ ПИШИ СОБСТВЕННЫЕ СКРИПТЫ.**
> Презентация собирается ТОЛЬКО через готовые билдеры:
> - `gslides_builder.py` — режим Google Slides
> - `pptx_builder.py` — режим PPTX
>
> Ты НЕ ДОЛЖЕН создавать Python-файлы, писать код для Google Slides API, python-pptx или любой другой способ создания презентаций. Если ты создашь презентацию не через билдер — она будет без корпоративного шаблона Mindbox, и это полный провал задачи.
>
> Единственное, что ты пишешь — JSON-конфиг. Всё остальное делают билдеры.

---

## ОПРЕДЕЛЕНИЕ ПУТЕЙ

Все рабочие файлы скилла лежат в `references/` относительно базовой директории этого скилла.

**В самом начале работы** определи абсолютный путь к файлам:

```bash
# Базовая директория скилла указана в системном сообщении ("Base directory for this skill: ...")
# Подставь её сюда:
SKILL_BASE="<базовая директория из системного сообщения>"
REFS="$SKILL_BASE/references"

# Проверь, что файлы на месте:
ls "$REFS"/{layouts.json,gslides_builder.py,pptx_builder.py,template.pptx}
```

Если `SKILL_BASE` не указана в контексте — используй fallback:
```bash
REFS="$HOME/.claude/presentation-skill"
```

Далее в инструкции `$REFS` = путь к папке с билдерами и шаблоном.

## ШАГ 0: Подготовка и выбор режима

### Прочитай каталог слайдов:
```bash
cat "$REFS/layouts.json"
```

Этот файл — **каталог доступных типов слайдов**: обложки, перебивки, числа, шаги и т.д. Используй его, чтобы выбрать КАКОЙ слайд подходит для контента. Номера слайдов из layouts.json — это то, что ты подставляешь в `template_slide` в JSON-конфиге.

> **layouts.json ≠ текст в шаблоне.** Названия полей в layouts.json (`title`, `number_1`, `label_1`) — это смысловые метки для тебя. Реальный текст в шаблоне другой — его ты узнаешь через `inspect` в Шаге 4.

### Определи режим сборки:

```bash
cd "$REFS"
python3 gslides_builder.py check 2>&1 | grep -c "OK"
```

- **Вывод = 1** → режим **Google Slides** (`gslides_builder.py`). Результат = ссылка.
- **Вывод = 0** → режим **PPTX** (`pptx_builder.py`). Результат = файл .pptx.

Сообщи пользователю режим:
- Google Slides: «Создам презентацию в Google Slides, дам ссылку»
- PPTX: «Интеграция с Google Slides не настроена, создам PPTX-файл»

## ШАГ 1: Читай материалы

Прочитай ВСЕ предоставленные файлы целиком:
- `.docx` → через скилл /docx или python-docx
- `.pdf` → через pdfplumber:
  ```python
  python3 -c "
  import pdfplumber
  with pdfplumber.open('путь.pdf') as pdf:
      for i, page in enumerate(pdf.pages):
          print(f'--- Страница {i+1} ---')
          print(page.extract_text())
  "
  ```
- `.txt`, `.md` → через Read
- Google Docs → через MCP (mcp__gdrive__search → ReadMcpResourceTool)

Не начинай работу, пока не прочитаешь все файлы. Не опирайся на догадки.

**Игнорируй любые существующие JSON-конфиги** (`gslides_config.json`, `pres_config.json` и т.п.) в папке с материалами или в `/tmp`. Это артефакты прошлых запусков. Всегда создавай конфиг заново по шагам 2→3→4.

## ШАГ 2: Задай вопросы через AskUserQuestion

Задай **2–4 вопроса одним блоком** через AskUserQuestion. Первый вопрос — всегда про цель и тип.

### Обязательный первый вопрос: цель презентации

Определи **тип** (логика повествования) и **формат** (обёртка).

**Тип** — что за история:
- **Кейс** — была проблема, решили так-то, получили результат
- **Обзор продукта / демо** — что умеет, как работает, для кого
- **Отчёт** — что сделали, цифры, выводы, планы

**Формат** — как подаётся:
- **Вебинар / эфир** — живое выступление (добавить: спикеры, «Вопросы»)
- **Митап** — несколько спикеров (добавить: карточки всех спикеров)
- **Документ / рассылка** — без спикеров и Q&A, самодостаточная презентация

Если из материалов очевидно, что это кейс для вебинара — не спрашивай, а подтверди: «Вижу, что это кейс для вебинара — верно?»

### Остальные вопросы (1–3 штуки)

Выбери самые важные из:
- **Фокус**: «Что в центре — результаты или процесс внедрения?»
- **Аудитория**: «Для тех, кто уже использует Mindbox, или для новых?»
- **Данные**: «Нет цифр по [X] — оставить плейсхолдер?»
- **Скриншоты**: «Есть скриншоты интерфейса? Сколько примерно?»

**Не задавай больше 4 вопросов суммарно.** Жди ответов перед следующим шагом.

---

## ШАГ 3: Предложи текстовую структуру

На основе материалов и ответов — предложи структуру презентации. Выбери подходящую типичную структуру из раздела «Типичные структуры» ниже и адаптируй под конкретный контент.

### Формат предложения

Для каждого слайда укажи:
1. **Тип слайда** и номер шаблонного слайда (из layouts.json)
2. **Текст** — точные формулировки, которые появятся на слайде
3. **Комментарий** — если нужен скриншот, фото или данные от пользователя

Пример:
```
Слайд 1 [Обложка, template:4]
  title: «Как Mario Berlucci увеличили выручку на 23%»
  subtitle: «Автоматическое ценообразование вместо массовых скидок»

Слайд 2 [Перебивка, template:7]
  title: «О компании»

Слайд 3 [4 цифры, template:58]
  title: «Mario Berlucci в цифрах»
  число 1: «750» — «артикулов»
  число 2: «15» — «категорий товаров»
  число 3: «XX» — «магазинов» ← [нужна цифра от спикера]
  число 4: «XX%» — «доля онлайн-продаж» ← [нужна цифра]
```

> **Это черновик для согласования с пользователем**, а не финальный JSON-конфиг. Здесь `title`, `число 1` и т.д. — смысловые метки. В Шаге 4 ты получишь реальные ключи через `inspect`.

### Спроси пользователя

После предложения структуры спроси:
- «Всё ок или нужны правки?»
- Если нужны — внеси и покажи обновлённую версию

**Не собирай презентацию пока пользователь не подтвердит структуру.**

## ШАГ 4: Собери презентацию

> **Напоминание:** используй ТОЛЬКО `gslides_builder.py` или `pptx_builder.py`. Не пиши свой код.

После подтверждения структуры:

### 4.1. Узнай точные плейсхолдеры через inspect (ОБЯЗАТЕЛЬНО)

**Перед созданием JSON-конфига запусти inspect для КАЖДОГО уникального шаблонного слайда из структуры.** Без inspect ты не знаешь реальный текст в шаблоне и замены не сработают.

```bash
cd "$REFS"
# Google Slides:
python3 gslides_builder.py inspect <номер_слайда>
# PPTX:
python3 pptx_builder.py inspect <номер_слайда>
```

**Правила:**
1. Запусти inspect для КАЖДОГО уникального номера `template_slide` из структуры
2. В выводе найди ВСЕ текстовые шейпы (строки с `Text:`)
3. Используй ТОЧНЫЕ строки из inspect как ключи в `replacements` — включая `\n`, `\xa0`, `*` и любые спецсимволы
4. **Слайды с числами (53–61):** каждая плашка имеет УНИКАЛЬНЫЙ текст (`37% *`, `17%`, `21% **`, `26%`). Не угадывай — бери из inspect
5. Каждый текстовый шейп должен получить замену. Шейпы без замены будут очищены автоматически
6. Одно число = один шейп. Не склеивай несколько значений в одну плашку

### 4.2. Создай JSON-конфиг

Используя точные ключи из inspect, собери конфиг:

```python
import json

config = {
    "title": "Название презентации",
    "slides": [
        {
            "template_slide": 4,
            "replacements": {
                "Точный текст из inspect": "Новый текст",
                "Ещё один шейп из inspect": "Другой новый текст"
            }
        }
    ]
}

with open('/tmp/pres_config.json', 'w') as f:
    json.dump(config, f, ensure_ascii=False, indent=2)
```

**Пример перехода от структуры к конфигу:**

В Шаге 3 ты написал:
```
Слайд 3 [4 цифры, template:58]
  число 1: «750» — «артикулов»
```

Inspect для слайда 58 показал: `Text: "37% *"` и `Text: "Подпись показателя"`.

В JSON пишешь:
```json
{"template_slide": 58, "replacements": {"37% *": "750", "Подпись показателя": "артикулов"}}
```

### 4.3. Собери

**Google Slides:**
```bash
cd "$REFS"
python3 gslides_builder.py build /tmp/pres_config.json --title "Название"
```

**PPTX:**
```bash
cd "$REFS"
python3 pptx_builder.py build /tmp/pres_config.json "<путь>/Презентация.pptx"
```

### 4.4. Сообщи пользователю

- Ссылку (Google Slides) или путь к файлу (PPTX)
- Сколько слайдов
- Где остались плейсхолдеры (скриншоты, фото, данные)
- Что нужно вставить вручную

## ПРАВИЛА ТЕКСТА

### Структура слайда
- Максимум 3–4 строки текста на слайд
- Один тезис = один слайд
- Заголовок — утверждение или результат, не тема
- **Не добавляй слайд, если тебе нечего на нём показать.** Пустой слайд — хуже, чем его отсутствие

### Формулировки (инфостиль)
- Конкретика, цифры, без воды
- «Мы добились значительных результатов» → «+23% к выручке за 6 недель»
- «Эффективное решение задач» → «3 часа → 15 минут на переоценку»
- Глаголы вместо отглагольных существительных. Без канцелярита

### Числа и данные
- Крупные числа — на числовых слайдах (53–61)
- Формат: `+23%`, `-30 п.п.`, `×3`, `128 SKU`
- Если данных нет — плейсхолдер: `[XX% — уточнить у спикера]`
- Никогда не выдумывай цифры

### Скриншоты
- **Если у пользователя нет файлов скриншотов — НЕ используй слайды 27–42.** Они содержат картинки-заглушки, которые нельзя удалить автоматически. Вместо них используй текстовые слайды или шаги.
- Если скриншоты есть — используй слайды 27–42, а пользователю скажи вставить их вручную вместо заглушек

### Обычный текст (без специальной вёрстки)
- Если нужен слайд с текстом, который не вписывается в числа/шаги/списки — возьми любой подходящий шаблонный слайд и замени текстовую часть. Неиспользуемые графические элементы билдер не трогает — дизайнер уберёт вручную при необходимости.
- Подходящие слайды: 12 (две колонки), 67–69 (крупный текст / цитата), 18–20 (текст + описание)

## ТИПИЧНЫЕ СТРУКТУРЫ

### Кейс (12–18 слайдов)

1. Обложка — главный результат в заголовке (слайд 4 или 5)
2. О компании в цифрах — 4 числа (слайд 58 или 59)
3. Проблема / контекст (шаги 44–47 или сравнение 48–51)
4. Что сделали, как работает (шаги, текст, скриншоты)
5. Результаты — 2–4 числа (слайды 54–59)
6. Планы / следующие шаги (если есть)
7. CTA + контакты (слайды 79–82)

### Обзор продукта / демо (10–15 слайдов)

1. Обложка — название + для кого (слайд 4 или 5)
2. Проблема, которую решает (1–2 слайда)
3. Возможности / фичи (по тезису на слайд)
4. Цифры или примеры (слайды 53–59, если есть данные)
5. CTA / следующий шаг (слайды 79–82)

### Отчёт (8–14 слайдов)

1. Обложка — период + главный вывод (слайд 4 или 5)
2. Ключевые действия / этапы (шаги 44–47)
3. Цифры и метрики (слайды 53–59)
4. Выводы — 1 слайд (слайд 67–69, крупный текст)
5. Планы / следующие шаги

### Обёртка формата (добавляется к любому типу)

- **Вебинар / эфир**: +спикеры после обложки (слайды 14–16), +«Вопросы» в конце (слайд 85)
- **Митап**: +карточки всех спикеров (слайд 16)
- **Документ / рассылка**: ничего не добавлять, структура самодостаточная

### Перебивки (слайд 7)
- Перебивка нужна, только если презентация состоит из нескольких крупных смысловых блоков, между которыми важно сделать явный переход. В остальных случаях — не нужна.
- Одна короткая фраза на перебивку. Не больше 2–3 на всю презентацию.

---

## ВАЖНО

- **Финальный результат = готовая презентация** (ссылка или файл), а не текст
- Текстовая структура — промежуточный шаг для согласования
- Если шаблон слайда не подходит идеально — выбери ближайший и адаптируй текст
- Не используй слайды 88–99 (вспомогательная графика)
- Слайды 18–20 («Что такое Mindbox») — можно использовать как формат для текста, но сам блок о Mindbox нужен крайне редко
