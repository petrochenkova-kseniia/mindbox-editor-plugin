# Внесение изменений в плагин mindbox-editor-plugin

Провести пользователя через весь процесс: от установки инструментов до создания Pull Request. Пользователи — редакторы, не разработчики. Объяснять просто, без лишних технических деталей.

## Главное правило

**НИКОГДА не пушить напрямую в main.** Все изменения — только через Pull Request. Это правило без исключений.

**НИКОГДА не изменять файлы плагина в директории `~/.claude/`.** Там хранятся кэшированные и служебные копии. Все изменения вносятся только в клонированный репозиторий и отправляются через Pull Request.

## Процесс внесения изменений

### Шаг 1. Проверить инструменты

Выполнить команды:

```bash
brew --version
gh --version
gh auth status
```

Если что-то не установлено или не настроено — обратиться к инструкции в **`references/setup-tools.md`**.

### Шаг 2. Скачать репозиторий

Попробовать найти локальную копию:

```bash
find ~ -maxdepth 4 -type d -name "mindbox-editor-plugin" -not -path "*/.claude/*" -exec test -d "{}/.git" \; -print 2>/dev/null
```

Если найдена — перейти в неё и к шагу 3.

Если не найдена — спросить пользователя, в какую папку скачать проект. Затем:

```bash
cd /путь/к/выбранной/папке
gh repo clone petrochenkova-kseniia/mindbox-editor-plugin
cd mindbox-editor-plugin
```

### Шаг 3. Подготовить ветку

Обновить main и создать новую ветку:

```bash
git checkout main
git pull origin main
git checkout -b feat/краткое-описание
```

Формат имени ветки:
- `feat/описание` — новый скилл или функциональность
- `fix/описание` — исправление
- `docs/описание` — документация

### Шаг 4. Определить тип изменений

Перед тем как вносить изменения, проанализировать запрос пользователя и определить куда попадёт каждая часть:

- **Новый скилл** — самостоятельная инструкция/процесс, который Claude будет выполнять по триггерной фразе
- **Изменение существующего скилла** — правка уже существующего файла в `skills/`
- **Общий контент** — стиль, политика, шаблон или справочный материал, который нужен нескольким скиллам (хранится в `shared/`)

Для общего контента определить подпапку в `shared/` по проекту. Подпапки именуются по типу проекта, например: `shared/Кейсы/`, `shared/Презентации/`, `shared/Посты/`. Сначала посмотреть какие подпапки уже есть в `shared/` и использовать существующие. Если контент не подходит ни к одной — спросить пользователя через AskUserQuestion, нужно ли создать новую папку и как её назвать.

Озвучить свой анализ: что к какой категории относится, в какую папку попадёт и почему. Затем спросить пользователя через AskUserQuestion, согласен ли он с распределением, и дать возможность скорректировать.

После подтверждения — внести изменения:

- **Новый скилл** — см. раздел «Как добавить скилл» ниже
- **Изменение существующего скилла** — найти файл в `skills/`, отредактировать
- **Общий контент** — добавить файл в `shared/<проект>/`. Из скилла ссылаться через относительный путь: `../../shared/<проект>/имя-файла.md`

### Шаг 5. Обновить версию плагина

Открыть `.claude-plugin/plugin.json` и поднять версию:
- Новый скилл → минорная версия (1.0.0 → 1.1.0)
- Исправление → патч (1.0.0 → 1.0.1)

**Без обновления версии изменения не дойдут до пользователей** — плагин кешируется по версии.

### Шаг 6. Закоммитить и отправить

```bash
git add .
git commit -m "feat(skills): краткое описание изменений"
git push -u origin имя-ветки
```

Формат коммитов: `тип(область): описание`
- `feat` — новая функциональность
- `fix` — исправление
- `docs` — документация

### Шаг 7. Создать Pull Request

Сформировать PR по шаблону из **`references/pr-template.md`**:

```bash
gh pr create --title "Краткое описание" --body "тело PR по шаблону"
```

После создания — дождаться проверки и одобрения. После мержа в main все пользователи получат обновление автоматически.

## Как добавить скилл

1. Создать директорию `skills/имя-скилла/`
2. Создать файл `skills/имя-скилла/SKILL.md`:

```markdown
---
name: имя-скилла
description: >
  Описание когда использовать этот скилл. Включить триггерные фразы:
  «фраза 1», «фраза 2», «фраза 3».
version: 0.1.0
---

# Название скилла

Содержимое скилла — инструкции, правила, примеры.
```

3. При необходимости добавить `references/` для материалов, специфичных для этого скилла
4. Если нужен контент, общий для нескольких скиллов — добавить его в `shared/` и ссылаться из SKILL.md через `../../shared/имя-файла.md`

### Правила для скиллов

- Имя директории — kebab-case (например, `tone-of-voice`)
- `name` в frontmatter совпадает с именем директории
- `description` содержит триггерные фразы — по ним Claude понимает когда активировать скилл
- Материалы только для одного скилла — в `references/` внутри скилла
- Материалы для нескольких скиллов — в `shared/` в корне плагина
- Весь контент на русском языке
